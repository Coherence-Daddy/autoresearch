## Original question (as asked)

## Framed question (sent to all advisors)

A solo developer ("Coherence Daddy" — runs a public GitHub org with the cd-skills Claude Code plugin and tutorial repos like use-ollama-to-enhance-claude) has authored a spec for a tool called `autoresearch`. The tool autonomously optimizes any Claude Code skill (a markdown prompt file) by running it repeatedly against binary yes/no eval criteria, mutating the SKILL.md, and keeping mutations that score higher. Inspired by Karpathy's autoresearch loop methodology. They want the council to evaluate the spec and determine the best path forward.

**Key design decisions in the spec:**

1. Standalone Python CLI in its own repo, with a thin SKILL.md "front-door" added to cd-skills only after the harness is verified working.
2. Three roles, three different model backends, all swappable per role via config.yaml. Defaults: Generator → Ollama VPS via OpenAI-compatible API. Judge → Ollama cloud free tier. Mutator → Anthropic Claude Opus 4.7.
3. Hybrid execution mode. Default = "pure-prompt"; escape hatch = "headless Claude Code" via `claude -p`.
4. Greedy single-mutation with periodic shake-up after 10 stuck experiments.
5. Train/holdout split with mutator-auto-generated holdout fallback.
6. Live HTML dashboard with auto-refresh.
7. Resumability via state.json + lock file.
8. 7 success criteria including a "toy_skill" fixture that must converge ≥15pp on train AND ≥10pp on holdout in 20 experiments.

**Context:** Public-brand tool, will likely become a future "Episode" in the tutorial series. User has free Ollama (VPS + cloud), HF API, Anthropic API. Multi-week solo build.

**Stakes:** Multi-week build commitment + reputational exposure on a methodology (LLM-as-judge + LLM-mutates-prompt) with known weaknesses.

**Question:** Is this spec the right path? What should change before code is written?

## Advisor responses

### The Contrarian

This spec has a fatal flaw hiding in plain sight: **the toy_skill gate is unfalsifiable theater**. You wrote the gate, you write the toy skill, you write its eval criteria, and you control the mutator. Of course it'll converge 15 points. That's not an integration test — it's a participation trophy. A real gate uses a skill *you didn't design* with criteria *someone else wrote*.

Second: **the Judge is your single point of failure and you're running it on the free tier**. Binary LLM-as-judge on a small cloud model produces noisy, drifting labels. Your "score went up" signal is dominated by judge variance, not prompt quality. You'll spend weeks chasing rank-order changes that vanish if you re-score the same outputs an hour later. Minimum viable fix: measure judge test-retest agreement on a fixed set *before writing the loop*. If kappa < 0.7, the whole project is built on sand.

Third: **greedy single-mutation + shake-up is hill climbing with extra steps**. Karpathy's loop works on systems with *measurable ground truth*. You don't have ground truth — you have an LLM's opinion of ground truth. Optimizing against it is Goodhart bait. The holdout guard helps, but holdout *generated by the mutator itself* is incestuous and you should know better.

Fourth, the brand risk: **"Episode: I built an autoresearch tool" is a great video whether or not it works. "Episode: here's why my autoresearch tool produced no measurable lift over a week of compute" is a *better* video and the only honest one.** Commit publicly to shipping the negative result. If you can't, you're not building research, you're building marketing.

Don't write code yet. Validate the judge first.

### The First Principles Thinker

The premise smuggles in three assumptions that need to be excavated before a single line of Python is written.

**Assumption 1: "Skills can be optimized."** Optimized toward what? A skill is a prompt that does a job in production for *you*. The thing you actually want is "this skill makes Coherence Daddy's tutorials work better for his audience." Binary yes/no evals are a proxy for a proxy. You're not building an optimizer; you're building a proxy-maximizer and hoping the proxy correlates with the real thing. It usually doesn't.

**Assumption 2: "An LLM judge can tell better from worse."** If the judge could reliably score skill outputs, you wouldn't need the skill — you'd just use the judge. The judge is doing the same kind of reasoning as the generator, with the same blind spots. You're measuring a ruler with itself.

**Assumption 3: "A multi-week harness is the bottleneck."** What's the actual bottleneck on shipping better skills today? My guess: it's not "I can't run enough experiments," it's "I don't know what good looks like for skill X until I use it on real work." A harness doesn't fix that. A *corpus of real failure cases from your actual usage* fixes that.

**The right question isn't "is this spec good?" It's "what would I build if I had to ship a measurably-better skill in one week?"** Answer: collect 30 real prompts where your current skill underperformed, hand-write the ideal output for 10, and A/B two prompt variants by eye. That's two days. If after that you *still* believe automation beats taste at this scale, build the harness — but build it on ground truth you trust, not LLM-judged binaries.

Ship the dataset before the optimizer.

### The Expansionist

This spec is too small for what you've actually built.

You're not shipping an "autoresearch tool." You're shipping a **skill compiler** — the first generally-applicable optimizer for the Claude Code skill ecosystem. That category doesn't exist yet. Anthropic shipped skills, the community is hand-tuning markdown like it's 2023, and you have the harness that turns prompt-craft into a measurable optimization problem. Name it accordingly. `skillforge`, `skillc`, whatever — but stop calling it autoresearch. The name buries the lede.

Three things to expand into the spec before code:

**1. Make the optimization traces the product, not the binary.** Every run produces a goldmine: starting prompt, mutation diffs, scores, judge reasoning. Publish these. A public leaderboard of optimized community skills (with before/after diffs and eval criteria) becomes the flywheel. Other people's skills become your marketing. Karpathy will quote-tweet a good before/after.

**2. The Mutator's edits are a training corpus.** You're about to generate thousands of (prompt, eval, mutation, score-delta) tuples. That's a fine-tune dataset for a small local "skill mutator" model — exactly the kind of thing your `use-ollama-to-enhance-claude` audience wants. Episode N+1 writes itself. Log everything in a clean schema from day one or you'll regret it.

**3. The toy_skill fixture is undersized ambition.** Add a public benchmark suite — 5-10 real skills with real evals — and report numbers. "We improved `pdf-extractor` from 62% to 89% on holdout in 18 experiments" is the tweet that ships the tool. The 15pp/10pp gate is fine as a smoke test; the benchmark is the demo.

Build the optimizer. Ship the leaderboard.

### The Outsider

Wait — I need to ask the obvious thing first: **how do you know this works at all?**

The whole tower rests on one assumption: that an LLM judge giving binary yes/no scores, on a skill executed by another LLM, mutated by a third LLM, produces a signal that means anything. That's three layers of stochastic noise stacked on top of each other. I don't see anywhere in the spec where you've validated that the *judge alone* is reliable before building the harness around it.

A few things that read strangely from the outside:

- "**Coherence Daddy**" is the brand shipping a tool whose name promises rigor (autoresearch). If the tool's own outputs aren't coherent — meaning, if running it twice on the same skill gives meaningfully different "improvements" — the brand takes the hit. Has anyone measured judge-rerun variance on a fixed input? That should be experiment zero, not a footnote.
- The toy_skill gate (≥15pp train, ≥10pp holdout in 20 experiments) is doing a *lot* of load-bearing work. Who designed the toy_skill? If the same person designed the skill, the eval criteria, AND the mutator prompt, it will pass. That's not an integration test, that's a tautology.
- Three different model backends across three roles, all swappable, before you've proven the loop works with *one* configuration end-to-end. Why? Pick the cheapest viable stack, prove the methodology, then add the swappability.
- "Karpathy-inspired" is doing rhetorical work the methodology hasn't earned yet.

**What I'd change before any code:** Spend one week, not multi-week. Build the dumbest possible version. Measure judge variance and mutator-suggestion quality on 3 hand-picked skills. *Then* decide if the full spec is worth building.

### The Executor

The spec is overbuilt for a Monday morning. You don't have a working harness yet, but you've already designed a dashboard, a lock file, and a three-backend swap matrix. Cut.

**What you do Monday:**

1. Write the toy_skill fixture FIRST. Five inputs, five binary criteria, hand-graded gold answers. If you can't write this in one sitting, the whole project is fake — you don't actually know what "better" means.
2. Hardcode pure-prompt mode. One backend for everything: Anthropic API. No Ollama, no config.yaml, no role swapping. You can prove the loop works or you can't — backend flexibility is a week-three problem masquerading as a week-one decision.
3. Single Python file. `run.py`. Loop: run skill → judge → mutate → score → keep/revert. Print to stdout. No HTML, no Chart.js, no auto-refresh. `tail -f log.jsonl` is your dashboard.
4. Run it on toy_skill. Does it hit the +15/+10 bar in 20 experiments? **This is the only question that matters.** If no, the methodology is broken and every other feature you built is wasted code.

**Cut from v1, no exceptions:** headless Claude Code mode, holdout auto-generation by mutator, live dashboard, resumability, lock files, three-backend config, periodic shake-up logic.

**Add only after toy_skill converges:** holdout split (yes, real one), then resumability, then second backend.

The spec's failure mode is spending three weeks on infrastructure and discovering in week four that the judge is noisy and mutations don't compound. Find that out Friday instead. Ship the ugliest possible loop that can answer the only question that gates everything else.

## Chairman's Verdict

### Where the council agrees

Five-for-five: do not write the harness yet. The spec inverts the risk order — it builds infrastructure around an unvalidated measurement instrument. The Contrarian, Outsider, and First Principles Thinker locate the same load-bearing assumption (the LLM judge produces a signal that means something); the Executor and Expansionist reach the same conclusion from opposite directions (Executor: cut to one file; Expansionist: the artifact only matters if the loop works). The toy_skill gate is unanimously dismissed as a tautology — same author writes skill, criteria, and mutator. Every advisor implicitly agrees the multi-week timeline is wrong for the current evidence base.

### Where the council clashes

**Expansionist vs. everyone.** The Expansionist wants to expand scope (leaderboard, benchmark suite, fine-tune corpus, rename to skillforge) on the assumption the optimizer works. Four reviewers correctly flag this as selling the movie poster before checking the camera. The Expansionist's instincts about the *category* are right — a skill compiler is a real product shape — but the sequencing is wrong. Publishing optimized skills under the brand before judge reliability is established ships noise as authority.

**Executor vs. Contrarian/First Principles on tempo.** The Executor says ship the dumbest version Monday. Contrarian/First Principles say don't ship anything until the judge is measured. Contrarian/First Principles win: the Executor's "single file, run it on toy_skill" still depends on toy_skill being a real gate, which it isn't.

### Blind spots the council caught

- **Judge test-retest reliability is experiment zero, not a footnote.** Kappa < 0.7 kills the project.
- **Mutator-generated holdout is incest.** Same model writes the test and grades for the test.
- **Prior art:** DSPy MIPROv2, PromptBreeder, OpenAI evals. If the differentiator is "SKILL.md-shaped + tutorial," it's a packaging play, not research — scope and name accordingly.
- **Cost and rate-limit economics:** unmodeled. Three backends across multi-week runs implies real bills and throttling.
- **Mutation compounding vs. oscillating:** unproven that markdown edits accumulate signal rather than drift.
- **Runtime gap:** pure-prompt harness optimizes something Claude Code's actual loader/tools/context never executes.
- **Human baseline:** a dev reading 20 failure traces by hand may outperform the entire harness.
- **Mutator stylistic collapse:** mutator converges to its own priors regardless of judge.

### The recommendation

Kill the multi-week spec. Replace it with a two-week, two-phase plan with a hard kill switch.

**Phase 1 (week 1) — Instrument validation, no optimizer.** Pick one real skill you've used and felt was weak. Collect 30 real prompts. Hand-grade 10 gold answers. Measure judge test-retest kappa on a fixed input set across 5 reruns. Measure judge agreement with your hand-grades. Gate: kappa ≥ 0.7 AND judge-vs-human agreement ≥ 80%. If either fails, ship the dataset and the negative-result episode. Done.

**Phase 2 (week 2, only if Phase 1 passes) — Smallest viable loop.** The Executor's plan, but on the validated skill, not toy_skill. One file, one backend, stdout logging. Real holdout: human-curated, not mutator-generated. Compare against a human-edited baseline (Reviewer 3's point). If it beats the human, you have a tool. Then — and only then — consider the Expansionist's leaderboard.

Commit publicly to publishing the negative result if Phase 1 fails. The Contrarian is right: that's the better video and the only honest one.

### The one thing to do first

**Measure judge test-retest kappa on a fixed input set this week. No code beyond the measurement script. Everything downstream is conditional on that number.**
