LLM Council Report

autoresearch — spec evaluation

Question brought to the council: evaluate the spec at docs/specs/2026-04-27-autoresearch.md and determine the best path. Spec proposes a standalone Python CLI that autonomously optimizes Claude Code skills via binary LLM-as-judge scoring + LLM-mutated SKILL.md, with three swappable model backends, hybrid execution modes, train/holdout split, and a live dashboard. Multi-week solo build with public-brand reputational stakes.

Chairman's Verdict

Where the council agrees

Five-for-five: do not write the harness yet. The spec inverts the risk order — it builds infrastructure around an unvalidated measurement instrument. The Contrarian, Outsider, and First Principles Thinker locate the same load-bearing assumption (the LLM judge produces signal that means something); the Executor and Expansionist reach the same conclusion from opposite directions. The toy_skill gate is unanimously dismissed as a tautology — same author writes skill, criteria, and mutator. Every advisor implicitly agrees the multi-week timeline is wrong for the current evidence base.

Where the council clashes

Expansionist vs. everyone. The Expansionist wants to expand scope (leaderboard, benchmark suite, fine-tune corpus, rename to skillforge) on the assumption the optimizer works. Four reviewers correctly flag this as selling the movie poster before checking the camera. The category instinct is right — a skill compiler is a real product shape — but the sequencing is wrong.

Executor vs. Contrarian/First Principles on tempo. The Executor says ship the dumbest version Monday. The Contrarian and First Principles say don't ship anything until the judge is measured. The latter wins: the Executor's "single file, run it on toy_skill" still depends on toy_skill being a real gate, which it isn't.

Blind spots the council caught (peer review)

  • Judge test-retest reliability is experiment zero, not a footnote. Kappa < 0.7 kills the project.
  • Mutator-generated holdout is incest. Same model writes the test and grades for it.
  • Prior art. DSPy MIPROv2, PromptBreeder, OpenAI evals. If the differentiator is "SKILL.md-shaped + tutorial," it's a packaging play, not research — scope and name accordingly.
  • Cost and rate-limit economics. Unmodeled. Three backends across multi-week runs implies real bills and throttling.
  • Mutation compounding vs. oscillating. Unproven that markdown edits accumulate signal rather than drift.
  • Runtime gap. Pure-prompt harness optimizes something Claude Code's actual loader/tools/context never executes.
  • Human baseline. A developer reading 20 failure traces by hand may outperform the entire harness.
  • Mutator stylistic collapse. Mutator converges to its own priors regardless of judge.

The recommendation

Kill the multi-week spec. Replace it with a two-week, two-phase plan with a hard kill switch.

Phase 1 (week 1) — instrument validation, no optimizer. Pick one real skill you've used and felt was weak. Collect 30 real prompts. Hand-grade 10 gold answers. Measure judge test-retest kappa on a fixed input set across 5 reruns. Measure judge agreement with your hand-grades. Gate: kappa ≥ 0.7 AND judge-vs-human agreement ≥ 80%. If either fails, ship the dataset and the negative-result episode. Done.

Phase 2 (week 2, only if Phase 1 passes) — smallest viable loop. Executor's plan, but on the validated skill, not toy_skill. One file, one backend, stdout logging. Real holdout: human-curated, not mutator-generated. Compare against a human-edited baseline. If it beats the human, you have a tool. Then — and only then — consider the leaderboard.

Commit publicly to publishing the negative result if Phase 1 fails. The Contrarian is right: that's the better video and the only honest one.

The one thing to do first

Measure judge test-retest kappa on a fixed input set this week. No code beyond the measurement script. Everything downstream is conditional on that number.

Advisor responses

The Contrarian Validate the judge first; the toy_skill gate is theater 5/5 named strongest

This spec has a fatal flaw hiding in plain sight: the toy_skill gate is unfalsifiable theater. You wrote the gate, you write the toy skill, you write its eval criteria, and you control the mutator. Of course it'll converge 15 points. That's not an integration test — it's a participation trophy. A real gate uses a skill you didn't design with criteria someone else wrote.

Second: the Judge is your single point of failure and you're running it on the free tier. Binary LLM-as-judge on a small cloud model produces noisy, drifting labels. Your "score went up" signal is dominated by judge variance, not prompt quality. Minimum viable fix: measure judge test-retest agreement on a fixed set before writing the loop. If kappa < 0.7, the whole project is built on sand.

Third: greedy single-mutation + shake-up is hill climbing with extra steps. Karpathy's loop works on systems with measurable ground truth. You don't have ground truth — you have an LLM's opinion of ground truth. Optimizing against it is Goodhart bait. The holdout guard helps, but holdout generated by the mutator itself is incestuous.

Fourth, brand risk: "Episode: I built an autoresearch tool" is a great video whether or not it works. "Episode: here's why my autoresearch tool produced no measurable lift" is a better video and the only honest one. Commit publicly to shipping the negative result. Don't write code yet. Validate the judge first.

The First Principles Thinker Ship the dataset before the optimizer

The premise smuggles in three assumptions that need to be excavated before a single line of Python is written.

Assumption 1: "Skills can be optimized." Optimized toward what? Binary yes/no evals are a proxy for a proxy. You're not building an optimizer; you're building a proxy-maximizer.

Assumption 2: "An LLM judge can tell better from worse." If the judge could reliably score skill outputs, you wouldn't need the skill — you'd just use the judge. You're measuring a ruler with itself.

Assumption 3: "A multi-week harness is the bottleneck." The actual bottleneck is "I don't know what good looks like for skill X until I use it on real work." A corpus of real failure cases from your actual usage fixes that. A harness doesn't.

The right question isn't "is this spec good?" It's "what would I build if I had to ship a measurably-better skill in one week?" Answer: collect 30 real prompts where your current skill underperformed, hand-write the ideal output for 10, and A/B two prompt variants by eye. Two days. If you still believe automation beats taste at this scale, build the harness — but build it on ground truth you trust, not LLM-judged binaries. Ship the dataset before the optimizer.

The Expansionist You're building a skill compiler — ship the leaderboard 5/5 named biggest blind spot

This spec is too small for what you've actually built. You're not shipping an "autoresearch tool." You're shipping a skill compiler — the first generally-applicable optimizer for the Claude Code skill ecosystem. That category doesn't exist yet. Name it accordingly. skillforge, skillc — but stop calling it autoresearch.

Three things to expand into the spec before code:

  1. Make the optimization traces the product, not the binary. Every run produces a goldmine: starting prompt, mutation diffs, scores, judge reasoning. A public leaderboard of optimized community skills (with before/after diffs and eval criteria) becomes the flywheel. Karpathy will quote-tweet a good before/after.
  2. The Mutator's edits are a training corpus. Thousands of (prompt, eval, mutation, score-delta) tuples = fine-tune dataset for a small local "skill mutator" model. Episode N+1 writes itself. Log everything in a clean schema from day one.
  3. The toy_skill fixture is undersized ambition. Add a public benchmark suite — 5-10 real skills with real evals — and report numbers. "We improved pdf-extractor from 62% to 89% on holdout in 18 experiments" is the tweet that ships the tool.

Build the optimizer. Ship the leaderboard.

The Outsider Three layers of stochastic noise — has anyone measured the judge?

Wait — how do you know this works at all? The whole tower rests on one assumption: that an LLM judge giving binary yes/no scores, on a skill executed by another LLM, mutated by a third LLM, produces a signal that means anything. Three layers of stochastic noise stacked on top of each other. I don't see anywhere in the spec where you've validated that the judge alone is reliable before building the harness around it.

A few things that read strangely from the outside:

  • The brand promises rigor; the tool's outputs may not be coherent across reruns. Has anyone measured judge-rerun variance on a fixed input? Should be experiment zero, not a footnote.
  • The toy_skill gate is doing a lot of load-bearing work. If the same person designed the skill, the eval criteria, AND the mutator prompt, it will pass. That's a tautology, not an integration test.
  • Three different model backends across three roles, all swappable, before you've proven the loop works with one configuration end-to-end. Pick the cheapest viable stack first.
  • "Karpathy-inspired" is doing rhetorical work the methodology hasn't earned yet.

Spend one week, not multi-week. Build the dumbest possible version. Measure judge variance and mutator-suggestion quality on 3 hand-picked skills. Then decide if the full spec is worth building.

The Executor Single file, one backend, run it Monday

The spec is overbuilt for a Monday morning. You've designed a dashboard, a lock file, and a three-backend swap matrix before there's a working harness. Cut.

Monday:

  1. Write toy_skill fixture FIRST. Five inputs, five binary criteria, hand-graded gold answers. If you can't write this in one sitting, you don't actually know what "better" means.
  2. Hardcode pure-prompt mode. One backend for everything: Anthropic API. No Ollama, no config.yaml, no role swapping.
  3. Single Python file. run.py. Loop: run skill → judge → mutate → score → keep/revert. Print to stdout. No HTML, no Chart.js. tail -f log.jsonl is your dashboard.
  4. Run on toy_skill. Hit +15/+10 in 20 experiments? This is the only question that matters.

Cut from v1: headless mode, holdout auto-generation, dashboard, resumability, lock files, three-backend config, shake-up logic.

Add only after toy_skill converges: real holdout split, then resumability, then second backend.

Failure mode: spending three weeks on infrastructure to discover in week four that the judge is noisy and mutations don't compound. Find that out Friday.

Peer review highlights

What the reviewers converged on

5 of 5 reviewers named the Contrarian as strongest. Common reasoning: only response with a measurable, falsifiable go/no-go gate (kappa ≥ 0.7) before code is written. Operationalizes a diagnosis that other advisors only describe.

5 of 5 reviewers named the Expansionist as biggest blind spot. Common reasoning: pricing the leaderboard before the camera works. If judge or proxy fails, every artifact the Expansionist wants to publish is contaminated exhaust under a brand that promises rigor.

4 of 5 reviewers flagged cost and rate-limit economics as the universal miss across all five advisors. Multi-week runs across three backends imply real bills and Anthropic/Ollama-cloud throttling. Spec's runtime economics are unexamined.

Other items only one reviewer caught each, but worth surfacing:

  • Prior art exists (DSPy MIPROv2, PromptBreeder, OpenAI evals). If the differentiator is SKILL.md packaging, scope it as a packaging play.
  • Mutation diversity collapse: mutator drifts to its own stylistic priors regardless of judge.
  • Runtime gap: pure-prompt harness may optimize something Claude Code's actual loader/tools/context never executes.
  • Human-in-the-loop baseline: a dev reading 20 failure traces and hand-editing might outperform the entire harness.
  • Whether markdown mutations actually compound vs. oscillate.
